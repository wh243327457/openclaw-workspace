# 子代理系统方案 v3（精简版）

## 设计原则

- **最小隔离**：只隔离必要的（记忆、微信、工作空间）
- **最大自由**：子代理和主代理能力相同，用户体验不受限
- **创建即复制**：从主代理当前状态复制技能和模板，个人内容不复制

---

## 风险分析：子代理出问题会影响主代理吗？

### 隔离保障

```
主代理                          子代理
─────────────                  ─────────────
进程: openclaw-gateway          进程: openclaw-gateway (独立)
工作空间: workspace/            工作空间: sub-agents/alice/workspace/
端口: 18789                     端口: 18800
微信: 你的微信                   微信: Alice 的微信
记忆: MEMORY.md                 记忆: MEMORY.md（独立副本）
配置: config.json               配置: config.json（独立副本）
```

**结论：不会影响。** 因为：
- 进程独立：子代理崩溃只影响自己
- 工作空间独立：子代理写坏数据只在自己目录里
- 技能已复制：子代理改了 skills 不影响主代理
- 微信独立：子代理微信出问题和主代理无关

### 唯一的共享点：registry.json

registry.json 在主代理的工作空间中，是唯一共享的文件。

**风险**：子代理如果通过某种方式修改了 registry.json（理论上不可能，因为目录隔离），会影响主代理。

**对策**：registry.json 由主代理独占写入，子代理不访问它。

### 真正可能出问题的场景

| 场景 | 影响主代理？ | 后果 | 恢复方式 |
|------|------------|------|---------|
| 子代理进程崩溃 | ❌ 不影响 | 子代理无响应 | 自动重启 |
| 子代理记忆写坏 | ❌ 不影响 | 子代理记忆混乱 | 从备份恢复记忆 |
| 子代理 skills 改坏 | ❌ 不影响 | 子代理行为异常 | 从主代理重新复制 skills |
| 子代理占用大量内存 | ⚠️ 可能 | 主代理变慢 | 资源限制 + 自动 kill |
| 子代理微信被封 | ❌ 不影响 | 子代理用户无法使用 | 重新绑定微信 |
| 子代理配置改乱 | ❌ 不影响 | 子代理行为异常 | 从主代理重新复制配置 |

## 回滚机制

### 创建快照

每次对子代理做重要操作前，自动创建快照：

```bash
# 创建快照
sh scripts/snapshot-subagent.sh alice
# → 生成 ~/.openclaw/sub-agents/alice/snapshots/2026-03-29-035700.tar.gz

# 恢复快照
sh scripts/restore-subagent.sh alice 2026-03-29-035700
```

### 什么时候自动创建快照

- 创建子代理后（初始快照）
- 更新子代理 skills 前
- 修改子代理配置前
- 用户通过微信修改 SOUL.md 前

### 回滚操作

```
你: "回滚小蓝到今天早上"
我: 
  1. 停止小蓝
  2. 恢复快照
  3. 重新启动小蓝
  4. "小蓝已回滚到 03:57 的状态"
```

### 极端情况：删除子代理重新创建

如果子代理彻底坏掉：

```
你: "删除小蓝，重新创建"
我:
  1. 备份当前状态（以防万一）
  2. 停止小蓝
  3. 删除工作空间
  4. 重新从主代理复制创建
  5. "小蓝已重新创建，之前的记忆已丢失"
```

主代理完全不受影响。

## 架构

```
主代理
├── workspace/              ← 你的完整工作空间
├── 可创建/管理子代理
├── 可读子代理的记忆和聊天
│
├── 子代理 A
│   └── workspace-alice/    ← 从主代理复制（不含个人内容）
│       ├── SOUL.md          ← 主代理模板
│       ├── AGENTS.md        ← 主代理模板
│       ├── IDENTITY.md      ← 空，由 Alice 自己填
│       ├── USER.md          ← 空，由 Alice 自己填
│       ├── MEMORY.md        ← 空，独立积累
│       ├── memory/          ← 独立
│       ├── skills/          ← 从主代理复制
│       ├── scripts/         ← 从主代理复制
│       ├── agent-team/      ← 从主代理复制
│       └── docs/            ← 从主代理复制
│
└── 子代理 B
    └── workspace-bob/
```

## 创建流程

主代理执行 "创建子代理" 时：

```
1. 生成独立的 gateway token
2. 分配端口（18800 起）
3. 复制主代理 workspace：
   ✅ skills/          （技能）
   ✅ scripts/         （脚本）
   ✅ agent-team/      （配置系统）
   ✅ docs/            （文档）
   ✅ SOUL.md          （模板）
   ✅ AGENTS.md        （模板）
   ✅ BOOTSTRAP.md     （初始化指南）
   ❌ MEMORY.md        （不复制，子代理自己积累）
   ❌ memory/          （不复制）
   ❌ IDENTITY.md      （不复制，子代理自己定义）
   ❌ USER.md          （不复制，子代理的用户自己填）
   ❌ TOOLS.md         （不复制）
4. 生成子代理的 gateway.toml
5. 配置微信 channel
6. 启动子代理
7. 注册到 registry
```

## 隔离边界

| 资源 | 主代理 | 子代理 |
|------|--------|--------|
| 工作空间 | ~/.openclaw/workspace/ | ~/.openclaw/sub-agents/{name}/workspace/ |
| 记忆 | 独立 | 独立 |
| 微信 | 独立绑定 | 独立绑定 |
| 技能 | 完整 | 从主代理复制 |
| 配置 UI | 端口 8090 | 端口 8090+N |

**只隔离这三样**：
1. 工作空间（目录隔离）
2. 记忆（独立 MEMORY.md + memory/）
3. 微信（不同微信号）

**不限制的**：
- exec 权限（和主代理一样）
- 网络访问（和主代理一样）
- 技能使用（完整继承）
- 模型选择（共享模型池）

## 用户体验

### 创建子代理（主代理操作）
```
你: "创建一个子代理，叫小蓝"
我: 
  1. 创建工作空间
  2. 复制技能和模板
  3. 生成配置
  4. 启动实例
  5. "小蓝已创建，请用微信扫码绑定"
  [显示二维码]
```

### 子代理用户（通过微信）
```
Alice: "你好"
小蓝: "你好！我是小蓝，有什么可以帮你的？"

Alice: "/名字"  
小蓝: "我还没有名字，你想叫我什么？"

Alice: "叫你小蓝"
小蓝: "好的，以后叫我小蓝 😊"

Alice: "帮我写个方案"
小蓝: [正常工作，和主代理一样]
```

### 管理子代理（主代理操作）
```
你: "小蓝的状态"
我: "运行中，端口 18800，绑定微信 Alice，最后活跃 5 分钟前"

你: "看看小蓝最近的聊天"
我: [显示小蓝的聊天记录]

你: "停止小蓝"
我: "已停止小蓝"
```

## 实现步骤

### 第一步：创建子代理脚本
- scripts/create-subagent.sh
- 复制工作空间（排除个人文件）
- 生成 gateway.toml
- 分配端口和 token

### 第二步：进程管理
- 子代理启动/停止脚本
- 健康检查（主代理定期检查）

### 第三步：微信绑定
- 子代理微信配置
- 用户配置指令处理

### 第四步：主代理管理
- 查看子代理状态
- 查看子代理聊天
- 停止/启动子代理

### 第五步：记忆读取
- 主代理可读子代理的 MEMORY.md 和 memory/
- 通过符号链接或定期同步实现
